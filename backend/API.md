# SCEMAS Backend API Documentation

**Backend stack**: Supabase (Auth, PostgREST, Realtime, Edge Functions) + AWS IoT Core  
**Frontend client**: `@supabase/supabase-js`

---

## Database Schema

These tables must exist in Supabase. AWS IoT → Lambda writes into `sensors` and `metric_history`.

```sql
-- Sensor devices and their latest readings
create table sensors (
  id           text primary key,          -- e.g. 's1'
  name         text not null,             -- e.g. 'DT-01'
  zone         text not null,
  status       text not null default 'active',  -- active | warning | offline
  aqi          numeric,
  noise        numeric,
  temperature  numeric,
  humidity     numeric,
  last_seen    timestamptz
);

-- Hourly aggregated metric history (populated by pg trigger or Lambda)
create table metric_history (
  id           bigint generated always as identity primary key,
  recorded_at  timestamptz not null default now(),
  zone         text,                     -- null = city-wide aggregate
  aqi          numeric,
  noise        numeric,
  temperature  numeric,
  humidity     numeric
);

-- Alerts generated by rule engine (pg function or Edge Function)
create table alerts (
  id            text primary key default gen_random_uuid()::text,
  sensor_name   text not null,
  zone          text not null,
  metric        text not null,           -- aqi | noise | temperature | humidity
  message       text not null,
  severity      text not null,           -- low | medium | high | critical
  status        text not null default 'active',  -- active | acknowledged | resolved
  triggered_at  timestamptz not null default now(),
  resolved_note text,
  resolved_by   uuid references auth.users(id),
  resolved_at   timestamptz
);

-- Alert rules (admin-managed)
create table alert_rules (
  id         text primary key default gen_random_uuid()::text,
  name       text not null,
  metric     text not null,
  operator   text not null,             -- gt | lt | gte | lte
  threshold  numeric not null,
  severity   text not null,
  enabled    boolean not null default true,
  zone       text                        -- null = all zones
);

-- User profiles (mirrors auth.users, extended with role/status)
create table user_profiles (
  id          uuid primary key references auth.users(id) on delete cascade,
  name        text not null,
  email       text not null unique,
  role        text not null default 'operator',  -- admin | operator | public
  status      text not null default 'active',
  last_login  timestamptz
);
```

### Row Level Security (RLS)

Enable RLS on every table. Roles are stored in `user_profiles.role` and also in `auth.users.raw_user_meta_data->>'role'` for JWT claims.

```sql
-- sensors: any authenticated user can read; nobody writes from client (IoT Lambda writes)
alter table sensors enable row level security;
create policy "authenticated read" on sensors for select using (auth.role() = 'authenticated');

-- metric_history: same
alter table metric_history enable row level security;
create policy "authenticated read" on metric_history for select using (auth.role() = 'authenticated');

-- alerts: operators and admins read/write
alter table alerts enable row level security;
create policy "operator read"   on alerts for select using (
  (auth.jwt() ->> 'user_role') in ('operator', 'admin')
);
create policy "operator update" on alerts for update using (
  (auth.jwt() ->> 'user_role') in ('operator', 'admin')
);

-- alert_rules: admin only
alter table alert_rules enable row level security;
create policy "admin all" on alert_rules for all using (
  (auth.jwt() ->> 'user_role') = 'admin'
);

-- user_profiles: admin read/write; each user can read their own row
alter table user_profiles enable row level security;
create policy "admin all"  on user_profiles for all using (
  (auth.jwt() ->> 'user_role') = 'admin'
);
create policy "self read"  on user_profiles for select using (
  auth.uid() = id
);
```

> To include `user_role` in the JWT, create a Supabase **Auth Hook** (Database Function) that adds `user_profiles.role` to the token claims.

## Authentication

Supabase Auth handles login/logout/refresh automatically via the JS client.

### Sign in

```ts
const { data, error } = await supabase.auth.signInWithPassword({
  email: 'alex.chen@city.ca',
  password: 'hunter2',
})
// data.user, data.session (contains access_token, refresh_token)
```

The session is persisted to localStorage automatically by `@supabase/supabase-js`. No manual `localStorage.setItem` needed.

To get the role after sign-in, read the user profile:
```ts
const { data: profile } = await supabase
  .from('user_profiles')
  .select('name, role')
  .eq('id', data.user.id)
  .single()
```

### Sign out

```ts
await supabase.auth.signOut()
```

### Check current session (on page load)

```ts
const { data: { user } } = await supabase.auth.getUser()
if (!user) router.push('/login')
```

### Auth state listener (optional, for route protection)

```ts
supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_OUT') router.push('/login')
})
```

## Sensors

All via PostgREST (auto-generated from the `sensors` table).

### Fetch all sensors

```ts
const { data: sensors } = await supabase
  .from('sensors')
  .select('*')
  .order('name')
```

### Fetch sensors filtered by zone

```ts
const { data } = await supabase
  .from('sensors')
  .select('*')
  .eq('zone', 'Industrial Zone')
```

### City-wide averages (active sensors only)

Either query client-side after fetching all sensors, or use a Postgres view:

```sql
create view city_averages as
select
  round(avg(aqi))         as aqi,
  round(avg(noise))       as noise,
  round(avg(temperature)) as temperature,
  round(avg(humidity))    as humidity,
  count(*)                as active_count
from sensors
where status != 'offline';
```

Then:
```ts
const { data } = await supabase.from('city_averages').select('*').single()
```

## Metric History (24-hour chart)

```ts
const since = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString()

const { data: chartData } = await supabase
  .from('metric_history')
  .select('recorded_at, aqi, noise, temperature, humidity')
  .is('zone', null)           // city-wide rows
  .gte('recorded_at', since)
  .order('recorded_at')
```

Format `recorded_at` to `HH:MM` on the frontend before passing to Recharts.

## Alerts

### Fetch all alerts

```ts
// Active + acknowledged (for AlertsTable)
const { data: activeAlerts } = await supabase
  .from('alerts')
  .select('*')
  .neq('status', 'resolved')
  .order('triggered_at', { ascending: false })

// All alerts (for AlertHistoryTable)
const { data: allAlerts } = await supabase
  .from('alerts')
  .select('*')
  .order('triggered_at', { ascending: false })
```

### Filter alerts (history tab filters)

```ts
let query = supabase.from('alerts').select('*')

if (filterStatus)  query = query.eq('status', filterStatus)
if (filterZone)    query = query.eq('zone', filterZone)
if (filterDate)    query = query.gte('triggered_at', filterDate)
                              .lt('triggered_at',  filterDate + 'T23:59:59')

const { data } = await query.order('triggered_at', { ascending: false })
```

### Acknowledge an alert

```ts
const { error } = await supabase
  .from('alerts')
  .update({ status: 'acknowledged' })
  .eq('id', alertId)
```

### Resolve an alert

```ts
const { error } = await supabase
  .from('alerts')
  .update({
    status:        'resolved',
    resolved_note: note || null,
    resolved_by:   (await supabase.auth.getUser()).data.user?.id,
    resolved_at:   new Date().toISOString(),
  })
  .eq('id', alertId)
```

## Alert Rules *(admin only — enforced by RLS)*

### Fetch all rules

```ts
const { data: rules } = await supabase
  .from('alert_rules')
  .select('*')
  .order('name')
```

### Create a rule

```ts
const { data: newRule, error } = await supabase
  .from('alert_rules')
  .insert({
    name:      'High AQI Alert',
    metric:    'aqi',
    operator:  'gt',
    threshold: 75,
    severity:  'high',
    enabled:   true,
    zone:      null,   // null = all zones
  })
  .select()
  .single()
```

### Update a rule (full replace)

```ts
const { error } = await supabase
  .from('alert_rules')
  .update({ name, metric, operator, threshold, severity, zone })
  .eq('id', ruleId)
```

### Toggle enabled

```ts
const { error } = await supabase
  .from('alert_rules')
  .update({ enabled: !currentEnabled })
  .eq('id', ruleId)
```

### Delete a rule

```ts
const { error } = await supabase
  .from('alert_rules')
  .delete()
  .eq('id', ruleId)
```

## Users *(admin only — enforced by RLS)*

### Fetch all users

```ts
const { data: users } = await supabase
  .from('user_profiles')
  .select('*')
  .order('name')
```

### Create a user

Creating a Cognito/Auth user requires the **service role key**, which must never be exposed to the browser. Use a **Supabase Edge Function**:

**Call from frontend:**
```ts
const { data, error } = await supabase.functions.invoke('create-user', {
  body: { name: 'Jane Smith', email: 'jane@city.ca', role: 'operator' },
})
```

**Edge Function** (`supabase/functions/create-user/index.ts`):
```ts
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'

Deno.serve(async (req) => {
  const adminClient = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  )

  const { name, email, role } = await req.json()

  // Create Cognito/Auth user (sends invite email with temp password)
  const { data: { user }, error: authErr } = await adminClient.auth.admin.createUser({
    email,
    email_confirm: true,
    user_metadata: { role },
  })
  if (authErr) return new Response(JSON.stringify({ error: authErr.message }), { status: 400 })

  // Insert profile row
  await adminClient.from('user_profiles').insert({
    id: user!.id, name, email, role, status: 'active',
  })

  return new Response(JSON.stringify({ id: user!.id, name, email, role }), {
    headers: { 'Content-Type': 'application/json' },
  })
})
```

## Realtime (Sensor & Alert Updates)

Supabase Realtime subscribes to Postgres change events, replacing the WebSocket connection.

### Live sensor updates (SensorsTab)

```ts
useEffect(() => {
  const channel = supabase
    .channel('sensors')
    .on(
      'postgres_changes',
      { event: 'UPDATE', schema: 'public', table: 'sensors' },
      (payload) => {
        setSensors(prev =>
          prev.map(s => s.id === payload.new.id ? { ...s, ...payload.new } : s)
        )
      }
    )
    .subscribe()

  return () => { supabase.removeChannel(channel) }
}, [])
```

### New alerts (admin/operator pages)

```ts
useEffect(() => {
  const channel = supabase
    .channel('new-alerts')
    .on(
      'postgres_changes',
      { event: 'INSERT', schema: 'public', table: 'alerts' },
      (payload) => {
        setAlerts(prev => [payload.new as Alert, ...prev])
      }
    )
    .subscribe()

  return () => { supabase.removeChannel(channel) }
}, [])
```

> Realtime must be **enabled** for each table in the Supabase dashboard under Database → Replication.

## AWS IoT Integration

AWS IoT Core has no direct Supabase connector. Use one of:

1. **IoT Rule → Lambda → Supabase REST** — Lambda calls `POST /rest/v1/sensors` with the service role key to upsert sensor readings.
2. **IoT Rule → HTTP Action** — direct HTTP POST to Supabase PostgREST endpoint (simpler, no Lambda needed for basic upserts).

The Lambda/action should upsert on sensor `id`:

```sql
insert into sensors (id, name, zone, aqi, noise, temperature, humidity, status, last_seen)
values (...)
on conflict (id) do update set
  aqi = excluded.aqi,
  noise = excluded.noise,
  temperature = excluded.temperature,
  humidity = excluded.humidity,
  status = excluded.status,
  last_seen = excluded.last_seen;
```

## Frontend Changes Required

### 1. Install the Supabase client

```bash
cd frontend
npm install @supabase/supabase-js
```

### 2. Environment variables — `frontend/.env.local`

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
```

### 3. Create `frontend/lib/supabase.ts`

See the file in the repo — it exports a singleton `supabase` client.

### 4. File-by-file changes

| File | What changes |
|------|-------------|
| `app/login/page.tsx` | Replace hardcoded demo auth → `supabase.auth.signInWithPassword()` |
| `app/admin/page.tsx` | Replace localStorage check → `supabase.auth.getUser()` + profile fetch; replace `logout()` → `supabase.auth.signOut()`; fetch alerts/rules/users from Supabase; add Realtime alert subscription |
| `app/operator/page.tsx` | Same auth changes; fetch alerts from Supabase; add Realtime subscription |
| `app/page.tsx` | Fetch `CITY_AVG`, `ZONES`, `CHART_DATA`, `SENSORS` from Supabase instead of static imports |
| `components/SensorsTab.tsx` | Fetch sensors on mount; subscribe to `sensors` Realtime channel |
| `components/AlertsTable.tsx` | `onAcknowledge` / `onConfirmResolve` call Supabase updates (handlers stay in parent page) |
| `components/AlertRulesTab.tsx` | `onCreate`, `onUpdate`, `onDelete`, `onToggle` call Supabase (handlers stay in parent page) |
| `components/UsersTab.tsx` | `onCreateUser` calls `supabase.functions.invoke('create-user', ...)` |
| `lib/data.ts` | Keep types; remove `INITIAL_*` mock constants once backend is live |

### 5. Role-based redirects

After sign-in, read `user_profiles.role` and redirect:
- `admin` → `/admin`
- `operator` → `/operator`
- anything else → `/` (public)
